# A gradually typed lambda calculus compiled to a coercion cast calculus

## Syntax

    Base types            ι          ::=  ℕ
    Types                 A,B,C,D    ::=  ι | ★ | A ⇒ B
    Ground types          G,H        ::=  ℕ | ★⇒★

    Contexts              Γ          ::=  · | Γ , (x : A)

    Surface terms         L,M,N      ::=  x
                                        | $ n
                                        | ƛ x:A. N
                                        | L ·[ ℓ ] M

    Coercions             c,d        ::=  id A
                                        | G !
                                        | G ? ℓ
                                        | c ↦ d
                                        | c ; d

    Cast terms            L,M,N      ::=  x
                                        | $ n
                                        | ƛ x:A. N
                                        | L · M
                                        | cast M [ c ]
                                        | blame ℓ

    Cast values           V,W        ::=  $ n
                                        | ƛ x:A. N
                                        | cast V [ G ! ]
                                        | cast V [ c ↦ d ]

    Evaluation frames     F          ::=  □ · M
                                        | V · □
                                        | cast □ [ c ]

    Notes.
      * These notes write variables using names and use capture-avoiding
        substitution `N[V/x]`.
      * The Agda mechanization is intrinsically typed and uses de Bruijn
        indices.
      * The Agda development stores the blame label `ℓ` as an argument of the
        application constructor. These notes write it explicitly as `·[ ℓ ]`.
      * The only ground function type is `★⇒★`. Projecting a dynamic value to a
        function first checks that its outer tag is the function tag, then
        recursively coerces arguments and results.

## Type Consistency

    Definition of the consistency relation:

      A ~ B

    The relation is inductively generated by the following rules:

    ----------------
    ℕ ~ ℕ

    ----------------
    ★ ~ ★

    ----------------
    ★ ~ ℕ

    ----------------
    ℕ ~ ★

    A ~ ★     ★ ~ B
    -------------------------
    ★ ~ (A ⇒ B)

    ★ ~ A     B ~ ★
    -------------------------
    (A ⇒ B) ~ ★

    C ~ A     B ~ D
    -----------------------------
    (A ⇒ B) ~ (C ⇒ D)

    Derived rules:

    ----------------
    ★ ~ A

    ----------------
    A ~ ★

    The surface typing rules use this consistency relation rather than type
    equality at applications. Consistency is symmetric, so for
    specification purposes it does not matter whether the function rule is written with
    the domain covariantly or contravariantly. We write the domain
    contravariantly because that makes the `coerce` function more convenient to
    define.

## Surface GTLC Typing

    Judgment:
      Γ ⊢g M : A

    -------------------------------
    Γ ⊢g x : A

    -------------------------------
    Γ ⊢g $ n : ℕ

    Γ , (x : A) ⊢g N : B
    -----------------------------------
    Γ ⊢g ƛ x:A. N : A ⇒ B

    Γ ⊢g L : A ⇒ B     Γ ⊢g M : A′     A′ ~ A
    ------------------------------------------------
    Γ ⊢g L ·[ ℓ ] M : B

    Γ ⊢g L : ★         Γ ⊢g M : A
    -----------------------------------
    Γ ⊢g L ·[ ℓ ] M : ★

    The first application rule applies a function whose domain is merely
    consistent with the argument type; the runtime cast needed to reconcile the
    two types is inserted by compilation. The second rule is for applying a
    dynamic term. Since the operator may fail a runtime function check, the
    result type remains `★`.

## Coercions

    Judgment:
      ⊢ c : A ⇨ B

    -----------------------
    ⊢ id A : A ⇨ A

    -------------------  (G ground)
    ⊢ G ! : G ⇨ ★

    ---------------------  (G ground)
    ⊢ G ? ℓ : ★ ⇨ G

    ⊢ c : C ⇨ A     ⊢ d : B ⇨ D
    -------------------------------------
    ⊢ c ↦ d : (A ⇒ B) ⇨ (C ⇒ D)

    ⊢ c : A ⇨ B     ⊢ d : B ⇨ C
    --------------------------------
    ⊢ c ; d : A ⇨ C

    Intuition:
      * `G !` injects a ground value into `★`.
      * `G ? ℓ` projects a dynamic value out of `★`, blaming label `ℓ` if the
        tag check fails.
      * `c ↦ d` casts functions by casting arguments with `c` and results with
        `d`.
      * `c ; d` is explicit sequencing of coercions.

### Coercions Generated From Consistency

    The compiler inserts coercions with a type-directed function
    `coerce(ℓ, p)`, where `p : A ~ B`:

    coerce(ℓ, ℕ ~ ℕ)               =  id ℕ
    coerce(ℓ, ★ ~ ★)               =  id ★
    coerce(ℓ, ★ ~ ℕ)               =  ℕ ? ℓ
    coerce(ℓ, ℕ ~ ★)               =  ℕ !

    coerce(ℓ, ★ ~ (A ⇒ B))
      =  (★⇒★ ? ℓ) ; (coerce(ℓ, A ~ ★) ↦ coerce(ℓ, ★ ~ B))

    coerce(ℓ, (A ⇒ B) ~ ★)
      =  (coerce(ℓ, ★ ~ A) ↦ coerce(ℓ, B ~ ★)) ; (★⇒★ !)

    coerce(ℓ, (A ⇒ B) ~ (C ⇒ D))
      =  coerce(ℓ, C ~ A) ↦ coerce(ℓ, B ~ D)

    Each generated coercion has the expected type:

      if p : A ~ B, then ⊢ coerce(ℓ, p) : A ⇨ B

## Cast Calculus Typing

    Judgment:
      Γ ⊢c M : A

    -------------------------------
    Γ ⊢c x : A

    -------------------------------
    Γ ⊢c $ n : ℕ

    Γ , (x : A) ⊢c N : B
    -----------------------------------
    Γ ⊢c ƛ x:A. N : A ⇒ B

    Γ ⊢c L : A ⇒ B     Γ ⊢c M : A
    --------------------------------
    Γ ⊢c L · M : B

    Γ ⊢c M : A     ⊢ c : A ⇨ B
    --------------------------------
    Γ ⊢c cast M [ c ] : B

    -------------------------------
    Γ ⊢c blame ℓ : A

    The cast calculus is an explicitly casted call-by-value lambda calculus. It
    has ordinary application typing; all graduality is mediated through
    explicit coercions.

## Compilation

    The source language does not reduce directly in this development. Its
    dynamic semantics is given by compiling a well-typed surface term to a cast
    calculus term and then reducing that target term.

    Compilation is defined on typing derivations:

    ⟦ x ⟧                         =  x
    ⟦ $ n ⟧                       =  $ n
    ⟦ ƛ x:A. N ⟧                  =  ƛ x:A. ⟦ N ⟧

    If Γ ⊢g L : A ⇒ B, Γ ⊢g M : A′, and p : A′ ~ A, then

    ⟦ L ·[ ℓ ] M ⟧
      =  ⟦ L ⟧ · cast ⟦ M ⟧ [ coerce(ℓ, p) ]

    If Γ ⊢g L : ★ and Γ ⊢g M : A, then

    ⟦ L ·[ ℓ ] M ⟧
      =  cast ⟦ L ⟧ [ coerce(ℓ, ★ ~ (★ ⇒ ★)) ]
           · cast ⟦ M ⟧ [ coerce(ℓ, A ~ ★) ]

    The first case inserts only the argument cast required by consistency. The
    second case makes dynamic application explicit by:
      1. projecting the operator from `★` to `★⇒★`, and
      2. injecting the argument into `★`.

    The compiled term is type preserving:

      if Γ ⊢g M : A, then Γ ⊢c ⟦ M ⟧ : A

## Cast Calculus Reduction

    One-step reduction:

      M —→ N

    The reduction rules are call-by-value and use the evaluation frames above.

    (ƛ x:A. N) · V
      —→ N[V/x]

    cast V [ id A ]
      —→ V

    cast V [ c ; d ]
      —→ cast (cast V [ c ]) [ d ]

    cast V [ c ↦ d ] · W
      —→ cast (V · cast W [ c ]) [ d ]

    cast (cast V [ G ! ]) [ G ? ℓ ]
      —→ V

    cast (cast V [ G ! ]) [ H ? ℓ ]
      —→ blame ℓ          when G ≢ H

    Congruence is given by frames:

    M —→ N
    -----------------------------
    plug(F, M) —→ plug(F, N)

    Blame propagates through every frame:

    -----------------------------
    plug(F, blame ℓ) —→ blame ℓ

    Operationally:
      * `β-id` removes redundant casts.
      * `β-seq` exposes sequential coercions one layer at a time.
      * `β-↦` delays function casts until application, casting the argument
        contravariantly and the result covariantly.
      * Projection succeeds only on injected ground values.
      * A failed ground check produces `blame ℓ`.

## Multi-step

    Reflexive-transitive closure:

      M —↠c M
      M —→ N and N —↠c P imply M —↠c P

    A closed well-typed GTLC program is evaluated by reducing its compilation
    `⟦ M ⟧` in this multi-step relation until it reaches either a cast-calculus
    value or `blame`.

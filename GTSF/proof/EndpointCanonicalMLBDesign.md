# Endpoint-Canonical MLB Design

File Charter:

- Purpose: record the endpoint-canonical maximal-lower-bound algorithm design
  for GTSF types.
- Scope: design decisions, algorithm outline, examples, and proof targets.
- Main dependency: the counterexample in `proof.MLBGlbCounterexample`, which
  shows that evidence-directed MLB coherence is too broad.

## Goal

The endpoint-canonical MLB algorithm should take endpoint types, not lower-bound
evidence:

```text
endpointMlb : Ty -> Ty -> Maybe Ty
```

The result should be canonical from the endpoints alone.  In particular, it
must not depend on the order in which a lower-bound proof happens to expose or
erase `forall` binders.

The motivating counterexample is:

```text
A = forall X. X -> *
B = forall Y. * -> Y
```

The arbitrary evidence-directed selector can choose either:

```text
forall X. forall Y. X -> Y
forall Y. forall X. X -> Y
```

The endpoint-canonical algorithm should choose the first one because the use of
`X` appears to the left of the use of `Y` in the aligned lower body.

## Executable Reference

The current executable reference implementation is
`endpoint_canonical_mlb.py`.  It is intentionally separate from the old
assumption-merging `mlb?` experiment while the Agda selector theorem is being
restated around endpoint-canonical order.

The regression and bounded model tests are in
`test_endpoint_canonical_mlb.py`.

The checked Agda implementation is `proof.EndpointCanonicalMLB`, with
computation-by-`refl` tests in `proof.EndpointCanonicalMLBTest`.

The intentionally simpler but slower reference algorithm is described in
`EndpointCanonicalMLBSimpleDesign.md`.  It replaces support-profile solving
with an exhaustive recursive search over the three supported `forall`
proof-rule combinations.  Future work should prove the efficient algorithm
equivalent to this exhaustive reference implementation.

## Core Design Decisions

### Input

The algorithm takes only two types as input.

It does not take:

- lower-bound evidence,
- `ImprecisionWf` derivations,
- selector routes,
- or `MlbTypeSelector` evidence.

Those objects belong in the proof of common-lower soundness, maximality, and
coherence, not in the canonical computation.

### Two Mutually Recursive Phases

The algorithm has two conceptual phases.

Phase 1 exposes local `forall` prefixes.

Given:

```text
forall X1. ... forall Xm. A0
forall Y1. ... forall Yn. B0
```

it records:

```text
left exposed binders:  X1, ..., Xm
right exposed binders: Y1, ..., Yn
left body:             A0
right body:            B0
```

Phase 1 does not decide whether any exposed binder is left-only, right-only, or
shared.  It only records which endpoint variables have come into scope and in
which exposure order.

Phase 2 descends through the exposed bodies and generates support constraints.
When it reaches a nested type position, such as under an arrow, it calls phase 1
again at that position.  Therefore the two phases are mutually recursive:

```text
mlbBlock A B = expose local forall prefixes, then call mlbBody
mlbBody env A B = descend structurally, calling mlbBlock at nested positions
```

### Support Profiles

A result binder has one fixed support profile with respect to the two
endpoints:

```text
(X, Y)  shared support for left X and right Y
(X, *)  left-only support for left X
(*, Y)  right-only support for right Y
```

The profile is generated by aligned variable positions in the endpoint bodies:

```text
X with Y  generates (X, Y)
X with *  generates (X, *)
* with Y  generates (*, Y)
```

Repeated uses of the same profile reuse the same result binder.  For example:

```text
forall X. X -> X
*
```

generates two uses of `(X, *)`, but the MLB is:

```text
forall Z. Z -> Z
```

not:

```text
forall Z. forall W. Z -> W
```

### Incompatible Profiles

An endpoint binder cannot be split across incompatible support profiles.

For example:

```text
forall X. X -> X
forall Y. Y -> *
```

generates:

```text
domain:   (X, Y)
codomain: (X, *)
```

This is inconsistent.  A shared assumption `Z <= Y` does not imply
`Z <= *`, and a left-only assumption `Z <= *` does not imply `Z <= Y`.
The algorithm should return `nothing`.

Likewise, one right binder cannot be matched with two different left binders:

```text
forall X. forall X'. X -> X'
forall Y. Y -> Y
```

would require both `(X, Y)` and `(X', Y)`, so it has no common lower bound.

### Unused Binders

Unused one-sided binders are failures.

For example:

```text
forall X. *
*
```

has no common lower bound.  The candidate `*` is not below
`forall X. *`, and the candidate `forall X. *` is not below `*` because
the erased source binder does not occur in the body.

Unused binders on both sides may be paired.

For example:

```text
forall X. *
forall Y. *
```

has MLB:

```text
forall Z. *
```

If there are multiple unused binders on both sides, pair them in exposure
order:

```text
forall X. forall X'. *
forall Y. forall Y'. *
```

has MLB:

```text
forall Z. forall W. *
```

Any leftover unused binder on just one side causes failure.

### Ordering Constraints

The solver builds an ordering graph over result binders.

The graph has two sources of required ordering constraints.

First, it preserves left exposure order.  If two profiles use left binders
`Xi` and `Xj`, and `Xi` is exposed before `Xj`, then the result binder for
`Xi` must appear before the result binder for `Xj`.

Second, it preserves right exposure order.  If two profiles use right binders
`Yi` and `Yj`, and `Yi` is exposed before `Yj`, then the result binder for
`Yi` must appear before the result binder for `Yj`.

If these constraints contain a cycle, the algorithm returns `nothing`.

For example:

```text
forall X. forall X'. X
forall Y. forall Y'. Y'
```

The body occurrence generates `(X, Y')`.  The unused binders would pair as
`(X', Y)`.  Left exposure order requires:

```text
(X, Y') < (X', Y)
```

Right exposure order requires:

```text
(X', Y) < (X, Y')
```

This cycle means there is no common lower bound.

### Tie Breaking

When the ordering graph does not force an order between two profiles, use a
deterministic tie breaker:

1. earlier first use in the generated lower body;
2. earlier left exposure index, when both profiles have left binders;
3. earlier right exposure index, when both profiles have right binders;
4. left-supported profiles before right-only profiles;
5. stable profile creation order.

This tie breaker is only for unconstrained pairs.  It must not override any
required exposure-order edge.  Making first-use a hard edge would incorrectly
reject examples such as:

```text
*
forall Y. forall Z. Z -> Y
```

where the right endpoint's exposure order must be preserved even though the
body occurrence order sees `Z` before `Y`.

## Algorithm Outline

### `mlbBlock`

`mlbBlock A B` performs one local quantifier step:

1. Strip the maximal local `forall` prefix from `A`.
2. Strip the maximal local `forall` prefix from `B`.
3. Record the exposed binders and their exposure order.
4. Call `mlbBody` on the exposed bodies.
5. Pair unused binders that remain unused on both sides.
6. Reject leftover unused one-sided binders.
7. Solve the profile-ordering graph.
8. Emit the sorted `forall` prefix around the generated lower body.

### `mlbBody`

`mlbBody` descends structurally.

Expected cases include:

```text
* with *             returns *
base i with base i   returns base i
base i with *        returns base i
* with base i        returns base i
arrow with arrow     recursively solves domain, then codomain
arrow with *         recursively solves both components against *
* with arrow         recursively solves * against both components
X with Y             records profile (X, Y)
X with *             records profile (X, *)
* with Y             records profile (*, Y)
```

When a recursive descent reaches another local type position, it calls
`mlbBlock`, so nested `forall` blocks use the same design.

## Examples

### The Bad GLB Pair

```text
A = forall X. X -> *
B = forall Y. * -> Y
```

Constraints:

```text
domain:   (X, *)
codomain: (*, Y)
```

The domain occurrence precedes the codomain occurrence, and there is no
conflicting endpoint exposure order.  The result is:

```text
forall Z. forall W. Z -> W
```

Nominally:

```text
forall X. forall Y. X -> Y
```

### Repeated One-Sided Use

```text
A = forall X. X -> X
B = *
```

Both positions generate `(X, *)`, so both use the same result binder:

```text
forall Z. Z -> Z
```

### Incompatible Use

```text
A = forall X. X -> X
B = forall Y. Y -> *
```

The generated profiles are `(X, Y)` and `(X, *)`.  These require incompatible
support for the same left binder `X`, so the result is:

```text
nothing
```

### Paired Unused Binders

```text
A = forall X. *
B = forall Y. *
```

The unused exposed binders are paired, giving:

```text
forall Z. *
```

### Crossing Exposure Order

```text
A = forall X. forall X'. X
B = forall Y. forall Y'. Y'
```

The used profile `(X, Y')` and unused profile `(X', Y)` force opposite orders
from the left and right endpoints, so the result is:

```text
nothing
```

## Proof Targets

The core soundness theorem should state:

```text
if A and B are well formed,
and endpointMlb A B = just C,
then C <= A and C <= B.
```

The maximality theorem must not state the GLB property:

```text
if A and B are well formed,
and endpointMlb A B = just C,
and D <= A and D <= B,
then D <= C.
```

That would say `C` is the greatest lower bound of `A` and `B`.  The checked
counterexample rules out using this as the general theorem shape.

The maximal-lower-bound theorem should instead state:

```text
if A and B are well formed,
and endpointMlb A B = just C,
and D <= A and D <= B,
and C <= D,
then D <= C.
```

Equivalently, there is no strictly larger common lower bound above `C`.  This
allows other maximal lower bounds to exist when they are incomparable with
`C`.

The failure-completeness theorem is useful but can be delayed:

```text
if A and B are well formed,
and endpointMlb A B = nothing,
then there is no D such that D <= A and D <= B.
```

The canonical coherence theorem should use the endpoint-canonical result:

```text
if A <= A',
and B <= B',
and endpointMlb A B = just C,
and endpointMlb A' B' = just C',
then C <= C'.
```

## Implementation Notes

- The first Agda implementation should stay endpoint-based.  Do not route it
  through `mlb-type` on arbitrary lower-bound evidence.
- In de Bruijn form, exposure order is the order in which local binders are
  stripped from the prefix.
- The generated body can use profile identifiers as placeholders, then the
  solver can close over the sorted profile list.
- The finite solver should reject cycles in the ordering graph.
- The proof should connect generated profiles to `ImprecisionWf` assumptions
  only after the endpoint-canonical type has been computed.
